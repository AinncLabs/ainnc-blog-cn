# Ainnc 博客内容长期规范

这份规范以 2026 年 7 月 10 日确认的“短、准、值得转发”为最高原则。外部旧文档中“每段至少 500 字”的要求已经废止，不得再次使用。

## 1. 篇幅与可读性

- 正文目标约 500–800 字，脚本允许 500–1200 字。
- 至少 3 个正文段落、8 句话和 2 个二级标题。
- 单个正文段落最多 320 字，优先控制在 80–180 字。
- 能用一句话说清楚的内容，不得换三种说法重复。

## 2. 每篇必须带走什么

- 一个明确且略带反直觉的中心判断，不写“XX 很重要”式废话。
- 一个数字、具体场景或可核对的例子；假设数据必须明确写成“假设”或“例如”。
- 一条 12–90 字的独立引用金句，脱离上下文仍然成立，适合截图或转发。
- 一组读者可以立即执行的动作，用“先做什么、检查什么、什么时候暂停”表达。
- 第一段直接回答问题或指出常见误区，不使用“随着行业发展”“众所周知”等空泛开场。

## 3. 语言与品牌语气

- 使用人话，句子尽量只表达一个意思。
- 禁用“依赖关系、执行前提、颗粒度、抓手、赋能、闭环、链路”等黑话。
- 产品介绍最多占正文一小段，并且只解释它如何帮助当前问题。
- 不写无法证明的“最好、绝对安全、行业领先”等结论。
- 比喻要帮助理解，不能只追求夸张。

## 4. 结构按内容需要选择

- 普通观点文章：反直觉开场 → 具体例子 → 金句 → 行动建议。
- 搜索问答文章：开头直接回答 → 关键判断 → 行动建议；必要时增加 2–4 条短 FAQ。
- 选型对比文章：先给结论 → 统一维度比较 → 诚实写短板 → 决策表或条件句；强调方案可以组合。
- TL;DR 不是机械必填项。复杂文章可使用 2–4 句 TL;DR；500 字短文优先用第一段和独立金句承担快速结论，避免重复正文。
- 表格只在三个以上维度确实更清楚时使用，不为了看起来专业强行加表格。

## 5. 敏感内容与案例

- 限流、封号和风控只讨论合规、暂停、排查和复盘，不提供规避平台检测的技巧。
- 客户案例必须脱敏；无法公开验证的数据改用明确的假设场景。
- 案例是论据，不得占据整篇文章。
- 涉及平台规则时避免写成永久不变的保证。

## 6. 发布前强制步骤

1. 运行 `npm run verify:blog`。
2. 校验失败时按具体原因修改，不得降低门槛迁就文章。
3. 运行 `npm run build`，确认 Astro 内容解析与所有页面生成成功。
4. 实际打开至少一篇新文章，检查段落留白、引用块和移动端可读性。
5. 新文章进入选题库前先与现有标题和角度查重。

校验脚本会自动检查文章总长度、句数、段落长度、重复句、具体信息、行动建议、二级标题、独立金句、禁用黑话和空泛开场。
